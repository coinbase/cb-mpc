# Demos

`demos/` contains multi-process, networked reference workflows that are intentionally
separate from the zero-setup `examples/` tree.

Run the commands below from `cb-mpc-go/` unless noted otherwise.

These demos may require:

- multiple processes or terminals
- local TLS or mTLS certificate generation
- explicit addresses and file paths

They are still demo code, not hardened production infrastructure or a
production-ready transport layer.

## ECDSA 2PC over mTLS

The first networked demo is [`demos/ecdsa2pc_mtls`](ecdsa2pc_mtls/). It runs two
separate processes that execute `ECDSA2PDKG`, `ECDSA2PSign`, and
`ECDSA2PRefresh` over a reference mTLS transport with explicit 2PC jobs.

The current mTLS helper is tuned for local demos:

- demo certificates generated by `scripts/certs/generate_certs.sh`
- peer identities embedded as deterministic names like `peer-alpha.demo.cbmpc.local`
- client-side verification against `localhost`

Applications using `cb-mpc-go` in production should generally plug in their own
networking and TLS implementation that matches their deployment, PKI, and
observability requirements.

The demo `--party-addrs` inputs are transport wiring only. They are not used as
MPC party names. The demos now use fixed UUID-form party identifiers inside
`Job2P` / `JobMP` to model the recommended practice for real deployments.

### Generate demo certs

```bash
make demo-certs
```

This writes PEM assets under `demos/tmp/certs/ecdsa2pc_mtls/`.
The generated peer directories use names like `peer-alpha/` and `peer-bravo/`.

### Run in two terminals

Terminal 1:

```bash
bash scripts/go_with_cpp.sh go run ./demos/ecdsa2pc_mtls/main.go \
  --self 0 \
  --listen 127.0.0.1:9440 \
  --ca demos/tmp/certs/ecdsa2pc_mtls/ca.pem \
  --cert demos/tmp/certs/ecdsa2pc_mtls/peer-alpha/cert.pem \
  --key demos/tmp/certs/ecdsa2pc_mtls/peer-alpha/key.pem
```

Terminal 2:

```bash
bash scripts/go_with_cpp.sh go run ./demos/ecdsa2pc_mtls/main.go \
  --self 1 \
  --peer 127.0.0.1:9440 \
  --ca demos/tmp/certs/ecdsa2pc_mtls/ca.pem \
  --cert demos/tmp/certs/ecdsa2pc_mtls/peer-bravo/cert.pem \
  --key demos/tmp/certs/ecdsa2pc_mtls/peer-bravo/key.pem
```

### Convenience target

For a single-command local run:

```bash
make demo-ecdsa2pc-mtls
```

## ECDSA MP Access-Structure over mTLS

[`demos/ecdsa_mp_ac_mtls`](ecdsa_mp_ac_mtls/) runs three signer processes over
the reference mTLS transport, performs `ECDSAMPDKGAC`, signs with a 2-of-3
online quorum, refreshes the AC key, and signs again with the refreshed key.

### Generate demo certs

```bash
make demo-certs DEMO_CERT_PARTIES=3 DEMO_CERT_DIR=demos/tmp/certs/ecdsa_mp_ac_mtls
```

### Run in three terminals

Shared addresses:

```bash
PARTY_ADDRS=127.0.0.1:9640,127.0.0.1:9641,127.0.0.1:9642
```

```bash
bash scripts/go_with_cpp.sh go run ./demos/ecdsa_mp_ac_mtls/main.go \
```

## ECDSA MP AC Web Demo

[`demos/ecdsa_mp_ac_web`](ecdsa_mp_ac_web/) runs three isolated Go web servers,
one per MPC party, and shows that `pkg/mpc` can back a browser-driven
application flow rather than just a one-shot CLI. Each party keeps its own local
state on disk, exchanges MPC messages over the reference mTLS transport, and
uses a fixed globally unique UUID-form party name inside `JobMP`.

This demo is also its own Go module so it behaves like a small consuming
application that imports `github.com/coinbase/cb-mpc-go`. The checked-in
`go.mod` uses a local `replace` to the `cb-mpc-go/` module root two directories
up; if you copy the demo out into another repository, remove or update that
`replace` and pin the `cb-mpc-go` version you want. The module keeps its own
demo transport and
workflow glue locally so it does not import `demos/common`; its `cb-mpc-go`
dependency surface is just `pkg/mpc`. The UI also exposes copyable joint public
key, local share, and receiver-side signature material to make the MPC flow
easier to inspect manually, including an uncompressed secp256k1 public key for
third-party verification tools that reject compressed keys.

Important caveats for this demo:

- the browser control plane is intentionally unauthenticated
- run it only on trusted localhost or similarly isolated development hosts
- per-party `data-dir` state includes plaintext sensitive MPC blobs such as key material and SIDs
- the `data-dir` layout is for usability demonstration only and is not a secure storage pattern
- the demo shows that an application can be built on `pkg/mpc`; it is not a hardened production template

Open one browser page per party server. On each page:

1. Create the same workflow ID and message.
2. Run DKG on all three pages.
3. On the participating pages only, choose the same sign quorum and run Sign.
4. Repeat Sign with another 2-of-3 quorum if you want to demonstrate flexible threshold signing.
5. Run Refresh on all three pages.
6. Choose any threshold quorum again and run Sign on the participating pages to demonstrate post-refresh signing.

The selected quorum's receiver slot is configured by `--sign-receiver` and
defaults to `0`, meaning the first party listed in the selected sign quorum
receives and verifies the DER signature.

The transport addresses are only for wiring the demo network. They are not used
as MPC party names.

### Generate demo certs

```bash
make demo-certs DEMO_CERT_PARTIES=3 DEMO_CERT_DIR=demos/ecdsa_mp_ac_web/tmp/certs
```

### Run in three terminals

Shared transport addresses:

```bash
REPO_ROOT=$(pwd)
DEMO_DIR="$REPO_ROOT/demos/ecdsa_mp_ac_web"
DEMO_CERT_DIR="$DEMO_DIR/tmp/certs"
DEMO_STATE_DIR="$DEMO_DIR/tmp/state"
TRANSPORT_ADDRS=127.0.0.1:9680,127.0.0.1:9681,127.0.0.1:9682
```

Party 0:

```bash
bash scripts/go_with_cpp.sh go -C demos/ecdsa_mp_ac_web run . \
  --self 0 \
  --http-listen 127.0.0.1:7080 \
  --transport-addrs "$TRANSPORT_ADDRS" \
  --data-dir "$DEMO_STATE_DIR/party-0" \
  --ca "$DEMO_CERT_DIR/ca.pem" \
  --cert "$DEMO_CERT_DIR/peer-alpha/cert.pem" \
  --key "$DEMO_CERT_DIR/peer-alpha/key.pem"
```

Party 1:

```bash
bash scripts/go_with_cpp.sh go -C demos/ecdsa_mp_ac_web run . \
  --self 1 \
  --http-listen 127.0.0.1:7081 \
  --transport-addrs "$TRANSPORT_ADDRS" \
  --data-dir "$DEMO_STATE_DIR/party-1" \
  --ca "$DEMO_CERT_DIR/ca.pem" \
  --cert "$DEMO_CERT_DIR/peer-bravo/cert.pem" \
  --key "$DEMO_CERT_DIR/peer-bravo/key.pem"
```

Party 2:

```bash
bash scripts/go_with_cpp.sh go -C demos/ecdsa_mp_ac_web run . \
  --self 2 \
  --http-listen 127.0.0.1:7082 \
  --transport-addrs "$TRANSPORT_ADDRS" \
  --data-dir "$DEMO_STATE_DIR/party-2" \
  --ca "$DEMO_CERT_DIR/ca.pem" \
  --cert "$DEMO_CERT_DIR/peer-charlie/cert.pem" \
  --key "$DEMO_CERT_DIR/peer-charlie/key.pem"
```

Then open:

- [http://127.0.0.1:7080](http://127.0.0.1:7080)
- [http://127.0.0.1:7081](http://127.0.0.1:7081)
- [http://127.0.0.1:7082](http://127.0.0.1:7082)

### Convenience target

```bash
make demo-ecdsa-mp-ac-web
```

Or run the module-local launcher directly:

```bash
bash demos/ecdsa_mp_ac_web/run_demo.sh
```

## TDH2 over mTLS

[`demos/tdh2_mtls`](tdh2_mtls/) runs three parties over the reference mTLS
transport, performs additive `TDH2DKGAdditive`, then has party 0 encrypt a
payload, broadcast the ciphertext to the other parties, collect their partial
decryptions, and recover the plaintext with `TDH2CombineAdditive`.

### Generate demo certs

```bash
make demo-certs DEMO_CERT_PARTIES=3 DEMO_CERT_DIR=demos/tmp/certs/tdh2_mtls
```

```bash
PARTY_ADDRS=127.0.0.1:9740,127.0.0.1:9741,127.0.0.1:9742
```

## Notes

- The reference mTLS transport now supports local full-mesh demo topologies by
  having each party listen on its own address and dial lower-index peers.
- The generated certificates are for local demos/tests only.
- These demos are reference workflows, not hardened production infrastructure.
